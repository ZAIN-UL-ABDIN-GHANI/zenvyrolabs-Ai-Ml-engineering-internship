REQUEST FLOW

Frontend to backend flow
1. frontend/index.html loads frontend/js/app.js.
2. The user enters a debate topic in the topic input field.
3. app.js sends POST /api/debate/start with JSON containing topic.
4. app.py receives the request and returns a JSON response with status and topic.
5. app.js renders the first message in the advocate feed and updates the status UI.
6. When the user presses the turn button, app.js sends POST /api/debate/next-turn with topic, last_speaker, and last_message.
7. app.py receives the request and currently returns a placeholder message.
8. The frontend falls back to local placeholder text if the backend does not provide a generated response.
9. The frontend continues alternating between agent panels as turn requests are made.

Machine-learning flow
1. The frontend does not call the ML routes directly in the current code.
2. app.py exposes POST /api/machine-learning/train and POST /api/machine-learning/evaluate.
3. /api/machine-learning/train calls DebateRegressionJudge.train_model("historical_debates.csv").
4. /api/machine-learning/evaluate is prepared to compare advocate_text and challenger_text.
5. The ML judge module is intended to extract features, train a regression model, and score arguments.

Demo-page flow
1. frontend/demo.html loads frontend/js/demo.js.
2. demo.js runs independently of the Flask backend.
3. It uses predefined arguments, a local timer, and a simulated judge overlay.
4. No API request is made from the demo flow.
